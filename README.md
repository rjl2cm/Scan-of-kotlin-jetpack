# 第一章：Android架构演进史

## 📜 Android架构演进对比表

| 阶段          | 架构模式           | 特点                                                                                                           | 问题                                                                                              | 优点                        | 缺点                          |
|---------------|--------------------|----------------------------------------------------------------------------------------------------------------|---------------------------------------------------------------------------------------------------|-----------------------------|-------------------------------|
| **早期阶段**  | **MVC**            | - View（视图层）：负责用户界面展示<br>- Model（数据层）：处理核心数据逻辑<br>- Controller（控制层）：调度业务逻辑 | - Activity 和 Fragment 承担大量逻辑，代码变得臃肿<br>- 难以进行单元测试<br>- 随规模增长难以维护   | - 架构简单<br>- 上手容易   | - 耦合性高<br>- 不适合复杂项目 |
| **中期阶段**  | **MVP**            | - 引入 Presenter 层分离 View 和 Model<br>- 通过接口进行 UI 解耦                                               | - Presenter 复杂度高<br>- Presenter 容易膨胀                                                    | - 解耦性增强<br>- 可测试性强 | - 需要更多接口定义            |
| **现代阶段**  | **MVVM (Jetpack)** | - ViewModel 管理 UI 状态<br>- LiveData 实现数据驱动<br>- 数据变化自动驱动 UI 更新                            | - 学习曲线较高<br>- 引入 Jetpack 组件库                                                          | - 数据驱动<br>- 模块化强    | - 组件依赖度变高              |
| **最新阶段**  | **现代架构模式**   | - Repository 模式<br>- 单向数据流<br>- 官方推荐架构组件（如 Navigation, DataBinding, WorkManager 等）          | - 多个模块之间可能存在复杂的依赖关系                                                             | - 提高开发效率<br>- 测试良好 | - 学习代码结构较复杂          |

---

## 说明

1. **架构演进：**
   - 从传统的 MVC 到现代的 Jetpack 架构，每一阶段的演进都在努力解决 Android 开发生态中代码的臃肿性、难以复用等硬伤。
   - 表格直观展示了每一阶段的技术特性和优缺点，希望读者能快速理解设计逻辑和现实问题。

2. **阅读提示：**
   - 如果您需要深入了解具体架构的详细内容，请参阅本文附录部分提供的相关资源。
